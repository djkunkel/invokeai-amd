# InvokeAI on the AMD Radeon AI PRO R9700 (gfx1201)

Scripts to run **[InvokeAI](https://github.com/invoke-ai/InvokeAI) 6.13.0** on an
**AMD Radeon AI PRO R9700** (RDNA4, reports as **gfx1201**) using **ROCm nightly**
PyTorch wheels, on an **immutable Fedora-family distro** (Bazzite / Fedora 44
base).

The R9700 is new enough that stable ROCm and stock PyTorch wheels don't yet
support it well. Getting a fast, working stack requires the ROCm *nightly*
multi-arch wheels (torch >= 2.12.0) plus a handful of platform workarounds. These
scripts wire all of that up inside a self-contained `.venv` — **no system ROCm is
installed**; everything (ROCm libs, torch, GPU kernels) lives in the venv, which
suits an immutable host where you can't freely layer system packages.

## Hardware / platform target

- **GPU:** AMD Radeon AI PRO R9700, `gcnArchName = gfx1201`, arch 12.0, ~32 GB.
- An integrated GPU (gfx1036) may also be present; `HIP_VISIBLE_DEVICES=0` hides it.
- **OS:** immutable Fedora-family (Bazzite / Fedora 44 base) — cannot `dnf install`
  system packages without layering, so workarounds are venv-local shims.
- Python 3.12, [`uv`](https://github.com/astral-sh/uv) for environment management.

## Quick start

```bash
# Full clean build: creates the venv, installs InvokeAI + the ROCm torch stack,
# and applies the bitsandbytes / libnuma shims.
./setup.sh

# Launch the web UI.
./run.sh
```

`run.sh` serves InvokeAI from `--root /mnt/extra/invokeai`; edit that path for
your install.

To verify the GPU stack independently of InvokeAI (tensor ops, flash attention,
patchmatch), run:

```bash
./test_torch.sh
```

## Scripts

| Script           | Purpose |
| ---------------- | ------- |
| `setup.sh`       | Full clean build: creates the venv, installs InvokeAI (CPU torch first), then runs `retorch.sh`, `bnb.sh`, `numa.sh`. |
| `retorch.sh`     | Installs the ROCm torch stack from the multi-arch nightly channel. `./retorch.sh` does a pinned install from `torch-versions.conf`; `./retorch.sh --defaults` (or `--latest`) installs whatever the channel currently resolves as latest (useful to probe new builds). |
| `bnb.sh`         | bitsandbytes ROCm shim — symlinks the bundled `libbitsandbytes_rocm72.so` to the name the current HIP version expects. |
| `numa.sh`        | libnuma shim — creates a venv-local unversioned `libnuma.so` (see below). |
| `run.sh`         | Launches `invokeai-web` with the required environment. |
| `test_torch.sh`  | Standalone torch/ROCm smoke test, incl. flash attention and patchmatch. |

## Configuration

- **`torch-versions.conf`** — the single source of truth for versions. Edit
  `NIGHTLY` (e.g. `rocm7.14.0a20260619`) plus the base versions `TORCH_VER` /
  `TORCHVISION_VER`. `retorch.sh` expands these into the uv overrides file.
- **`.torch-overrides.generated.txt`** — auto-generated by `retorch.sh`,
  gitignored. Do not edit or commit.

### Iterating on a new nightly

1. Edit `NIGHTLY` (and base versions if they changed) in `torch-versions.conf`.
2. Run `./retorch.sh`. Its two sanity checks (compute kpack + aotriton flash
   images) must pass.
3. Run `./test_torch.sh`; confirm the FLASH/EFFICIENT backends PASS and the speed
   line shows flash clearly faster than MATH.
4. To discover the latest available versions/packages, use `./retorch.sh --defaults`.

Use the **multi-arch** channel (`https://rocm.nightlies.amd.com/whl-multi-arch/`).
Do **not** mix in the per-arch `v2/gfx120X-all/` channel — its ROCm packaging is
incompatible with the multi-arch packages and mixing them segfaults.

## Why this is fiddly (hard-won gotchas)

This stack has four separate gotchas; all must hold or generation breaks.
`retorch.sh` sanity-checks the first two, and `test_torch.sh` exercises all four.

1. **Compute kernels** come from `amd-torch-device-gfx1201`
   (`torch/.kpack/torch_gfx1201.kpack`), loaded at runtime via the
   `.rocm_kpack_ref` section in `libtorch_hip.so`. Missing → every GPU op fails
   with `HIP error: device kernel image is invalid`.

2. **Flash / mem-efficient attention kernels** come from the gfx12 **family**
   package `amd-torch-device-gfx12-0` (NOT the gfx1201-specific one), which ships
   `torch/lib/aotriton.images/amd-gfx120x/*.aks2`. These exist only in
   **torch >= 2.12.0**. Missing → flash SDPA crashes with `HIP error: invalid
   argument` and silently falls back to ~8x slower MATH. Requires
   `TORCH_ROCM_AOTRITON_ENABLE_EXPERIMENTAL=1` at runtime (set in `run.sh` /
   `test_torch.sh`) — gfx1201 aotriton support is experimental.

3. **libnuma:** `libtorch_rocshmem.so` does `dlopen("libnuma.so")` (unversioned),
   but Fedora's `numactl-libs` ships only `libnuma.so.1`. `numa.sh` creates a
   venv-local `libnuma.so` symlink and `run.sh` / `test_torch.sh` add it to
   `LD_LIBRARY_PATH`. Without it you get a harmless-but-noisy "rocSHMEM Could not
   open libnuma" notice.

4. **FlexiBLAS vs ROCm OpenBLAS** (patchmatch): InvokeAI's patchmatch infill
   links Fedora's OpenCV stack and thus `libflexiblas.so.3`. With the default
   `openblas-openmp` backend, FlexiBLAS's load-time init dispatches into torch's
   already-loaded ROCm `librocm-openblas.so.0` and **segfaults during library
   load**, before any inference. Fix: `export FLEXIBLAS=netlib` (set in `run.sh` /
   `test_torch.sh`), which avoids the OpenBLAS dispatch path. This only affects
   the CPU patchmatch path; GPU/flash kernels are unchanged.
