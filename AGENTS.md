# AGENTS.md

Guidance for AI agents (and humans) working in this repo.

## What this is

A set of scripts to run **InvokeAI 6.13.0** on an **AMD Radeon AI PRO R9700**
(RDNA4, reports as **gfx1201**) using **ROCm nightly** PyTorch wheels, on an
**immutable Fedora-family distro (Bazzite / Fedora 44 base)**. No system ROCm is
installed; everything (ROCm libs, torch, kernels) lives in the `.venv`.

## Hardware / platform facts

- GPU: AMD Radeon AI PRO R9700, `gcnArchName = gfx1201`, arch 12.0, ~32GB.
- An integrated GPU (gfx1036) is also present; `HIP_VISIBLE_DEVICES=0` hides it.
- Immutable distro: cannot `dnf install` system packages without layering.
- The venv and the uv cache are both on `/var/mnt/extra` (same filesystem), so
  uv hardlinks work; no `UV_LINK_MODE=copy` needed.

## Scripts

- `setup.sh` — full clean build: creates the venv, installs InvokeAI (CPU torch
  first), then runs `retorch.sh`, `bnb.sh`, `numa.sh`.
- `retorch.sh` — installs the ROCm torch stack from the multi-arch nightly
  channel. Two modes:
  - `./retorch.sh` — pinned install from `torch-versions.conf`.
  - `./retorch.sh --defaults` (or `--latest`) — no version pins; installs
    whatever the channel resolves as latest (use to probe new builds).
- `bnb.sh` — bitsandbytes ROCm shim (symlinks `libbitsandbytes_rocm72.so` to the
  current HIP version's expected name).
- `numa.sh` — libnuma shim (see below).
- `run.sh` — launches `invokeai-web`.
- `test_torch.sh` — standalone torch/ROCm smoke test incl. flash-attention.

## Config files

- `torch-versions.conf` — the **single source of truth** for versions. Edit
  `NIGHTLY` (e.g. `rocm7.14.0a20260619`) plus base versions `TORCH_VER` /
  `TORCHVISION_VER`. `retorch.sh` expands these into the uv overrides file.
- `.torch-overrides.generated.txt` — auto-generated by `retorch.sh`, gitignored.
  Do not edit or commit.

## Critical knowledge (hard-won; do not regress)

This stack has four separate gotchas. All must hold or generation breaks.
`retorch.sh` has sanity checks for the first two; `test_torch.sh` exercises all
four.

1. **Compute kernels** come from `amd-torch-device-gfx1201`
   (`torch/.kpack/torch_gfx1201.kpack`). torch loads them at runtime via the
   `.rocm_kpack_ref` section in `libtorch_hip.so`. Missing → every GPU op fails
   with `HIP error: device kernel image is invalid`
   (`kpack_load_code_object failed with error: 13`).

2. **Flash / mem-efficient attention kernels** come from
   **`amd-torch-device-gfx12-0`** — the gfx12 **family** package, NOT the
   gfx1201-specific one. It ships `torch/lib/aotriton.images/amd-gfx120x/*.aks2`.
   - These images only exist in **torch >= 2.12.0** multi-arch wheels (2.10.0
     shipped libaotriton without them).
   - Missing → flash/mem-efficient SDPA decompresses to a NULL kernel and
     crashes with `HIP error: invalid argument`
     (`hipModuleLaunchKernel(<null>, ...)`), silently forcing the ~8x slower
     MATH SDPA fallback (unusable for the image diffusion transformer at real
     resolutions).
   - Requires `TORCH_ROCM_AOTRITON_ENABLE_EXPERIMENTAL=1` at runtime (set in
     `run.sh` and `test_torch.sh`) — gfx1201 aotriton support is experimental.

3. **libnuma**: `libtorch_rocshmem.so` does `dlopen("libnuma.so")` (unversioned).
   Fedora's `numactl-libs` ships only `libnuma.so.1`; the unversioned symlink is
   in `numactl-devel`, which can't be layered easily on immutable distros.
   `numa.sh` creates `.venv/numa-shim/libnuma.so -> libnuma.so.1` and
   `run.sh`/`test_torch.sh` add that dir to `LD_LIBRARY_PATH`. Without it you get
   a harmless-but-noisy `rocSHMEM Could not open libnuma` notice (rocSHMEM is
   unused here, so it's cosmetic).

4. **FlexiBLAS vs ROCm OpenBLAS** (patchmatch): InvokeAI's patchmatch infill
   loads `patchmatch/libpatchmatch.so`, which links Fedora's system OpenCV stack
   and thus `libflexiblas.so.3`. At `dlopen()` time FlexiBLAS's initializer picks
   a backend BLAS; with the default `openblas-openmp` it dispatches into the
   ROCm-bundled `librocm-openblas.so.0` (already loaded by torch) and **segfaults
   in `ztrsm_iunncopy_COOPERLAKE`** — coredump during plain library load, before
   any inference. Fix: `export FLEXIBLAS=netlib` (set in `run.sh` and
   `test_torch.sh`), which avoids the OpenBLAS dispatch path. patchmatch then
   loads and inpaints correctly. This only triggers when torch is imported
   *before* patchmatch (i.e. always, in invokeai-web). The old workaround was
   `INVOKEAI_PATCHMATCH=false`, which disabled patchmatch entirely. Note
   `FLEXIBLAS=netlib` only affects this CPU patchmatch path; GPU/flash kernels
   are unchanged (test still shows ~8x flash speedup).

### Packaging-trap pattern

The multi-arch channel splits torch's GPU kernels across multiple packages. The
`[device-gfx1201]` extra is supposed to pull them transitively, but combining
that extra with `--overrides` **drops the transitive deps**. So `retorch.sh`
installs the device packages **explicitly**. If flash or compute kernels go
missing again, suspect a new split-out package that isn't being listed
explicitly.

## Channel notes

- Use the **multi-arch** channel: `https://rocm.nightlies.amd.com/whl-multi-arch/`.
- Do **NOT** use the per-arch `v2/gfx120X-all/` channel. It bundles aotriton
  images too, but its ROCm packaging is incompatible with the multi-arch
  packages and mixing them segfaults; it's also less maintained.
- Only the `rocm7.14.0` series exists on multi-arch right now.

## How to iterate on a new nightly

1. Edit `NIGHTLY` (and base versions if they changed) in `torch-versions.conf`.
2. Run `./retorch.sh`. Its two sanity checks must pass (kpack + aotriton images).
3. Run `./test_torch.sh`; confirm FLASH/EFFICIENT backends PASS and the speed
   line shows flash clearly faster than MATH.
4. To discover the latest available versions/packages, use `./retorch.sh --defaults`.

## Conventions

- Don't commit `.venv/` or `.torch-overrides.generated.txt` (gitignored).
- Keep changes to the install flow reflected in `retorch.sh` sanity checks and
  `test_torch.sh` coverage.
- Prefer self-contained venv shims (like `bnb.sh`/`numa.sh`) over system
  packages, since the host is immutable.
